# When I reject AI code even if it works

# 即使AI代码能运行，我也会拒绝它

随着代码实现速度越来越快，真正的瓶颈转移到了审查AI生成的大量代码上。我说的还不是同事们（以及他们的智能体）提交的PR，而是你自己的编程智能体完成任务后产生的 `git diff`。

即使我遵循了良好的实践——比如从计划模式开始、将大任务分成多个阶段、提交小改动——在审查那些我自己并未真正思考过的内容时，我仍然感到认知过载。

在编程智能体出现之前，面对一项任务，我会先探索代码库、思考不同的解决方案、进行实验，然后才开始实现。这个过程可能需要数天来整合所有上下文。当我最终提交PR时，信心更高，向同事解释每一处改动也更容易。

我必须承认，即使有了AI，完成大型任务仍然需要我数天时间。很多时候，我会拒绝AI所做的所有改动，然后从头开始。第一次会话和第二次会话之间的区别不在于大语言模型，而在于屏幕后面的人。有了更多时间来梳理我要解决的问题，我就能引导智能体找到更好的方案，而不是被它牵着走。

越来越多时候，我拒绝AI代码的原因都是相同的：

- 当我无法用自己的话解释这个方案时，我拒绝AI代码。
- 当改动范围大于问题本身时，我拒绝AI代码。
- 当它在证明必要性之前就引入抽象时，我拒绝AI代码。
- 当它在本地能运行但让系统更难理解时，我拒绝AI代码。
- 当我对输出的信任超过了自己的理解时，我拒绝AI代码。

看到工程师过快地接受AI生成的改动并不罕见，这也是为什么我主张在AI审查的基础上仍然需要人工审查。现实是，能运行且让CI变绿的代码仍然可能是一个糟糕的方案，而工程的本质始终是实现适当的、可扩展的、可延展的解决方案。

我使用编程智能体已有一段时间，尽管它们令人印象深刻，但它们仍然需要优秀的工程师来引导它们产出优秀的方案。是的，编程智能体在这项任务上能帮助你的不仅仅是写代码，但这并不意味着它们目前能以可持续的方式自主完成这些工作。
