# Epoll vs. io_uring in Linux

# Linux 中的 epoll 与 io_uring

首先，我想告诉你我是怎么走到这一步的，以及为什么我开始研究 Linux 上处理异步 I/O 的不同方案……去年，我和学生们一起构建了一个名为 TinyGate 的反向代理服务器。它非常简单，基于 worker 模型，基本上运行良好。当然，我并不指望它很快，但这是一个教学项目，而且我们做出了一个真正的、算是可以投入生产的工具，所以我对此非常自豪。但我的学生们并不像我那样高兴——他们想构建真正有用的东西，对于我们的"产品"存在严重的架构限制、无法超越 nginx 和 haproxy 这样的巨头而感到非常失望。所以他们几乎是逼着我一起研究这些工具底层是怎么运作的，以及如何处理异步 I/O 来减少巨大的开销……长话短说，我们基于 epoll 做了 TinyGate 的第二个版本。在基准测试中它仍然输给了 nginx/haproxy，但与第一个版本相比，性能有了巨大的提升。但 epoll 也并不完美（下文会解释），我们最终切换到了 io_uring，这导致我们的项目又一次从头重写……所以这是一个非常有趣的话题，今天我将分享 Linux 为异步 I/O 提供的两种队列系统的概述。

## epoll 的历史

当我刚开始做 Linux 开发时，epoll 还是一个新功能，基本上没有替代方案。每个人都用它来管理异步执行——没有其他选择。问题是，epoll 严重依赖系统调用：它告诉你 I/O 何时可以进行，但之后你仍然需要自己调用 read()/write()——每个 I/O 事件需要两次系统调用，加上一次性的 epoll_ctl 注册。每一次系统调用都会导致用户态和内核态之间的上下文切换，当你处理大量连接时，这会产生巨大的开销。但我们有解决方案！在 epoll 进入 Linux 内核（2002年）大约 17 年后，io_uring 出现了（2019年）！它不是告诉你 I/O 何时可以进行，而是告诉你 I/O 何时完成——不需要轮询循环，关联的系统调用也少得多。

内核从你的应用程序和内核之间共享的内存中消费提交请求，并将完成结果写回同一块共享内存——两者都存在于环形缓冲区中，这就是名称的由来。需要注意的是：默认情况下你仍然需要调用 `io_uring_enter()` 来告诉内核"去检查提交队列"——但一次调用就可以提交一整批操作并收割一整批完成结果，而不是像 epoll + read 那样每个操作一对系统调用。如果你希望在稳态运行时接近零系统调用，可以使用 `IORING_SETUP_SQPOLL`，它会启动一个专用的内核线程来为你轮询提交队列——代价是该线程会消耗 CPU（下文详述）。

## 简单对比

基本架构：如前所述，epoll 在 I/O 可以进行时通知你，io_uring 在 I/O 完成时通知你。epoll 让每个 I/O 操作都要跨越内核边界，而 io_uring 让你只需支付一次较小的"设置费"（创建环）加上每批次的费用（`io_uring_enter()` 调用），而不是每个操作的费用。所以你得到的不是每次 I/O 一对系统调用，而是每批 I/O 一次系统调用——或者使用 SQPOLL 的话，几乎没有系统调用。可以看到，在大量 I/O 发生时，这节省了大量系统调用。

在支持 io_uring 的较新系统上（内核 v5.1+，2019年发布），通常没有太多理由再去使用 epoll。从就绪模型到完成模型的转变是一个巨大的架构变化——它把很大一部分工作从你的应用程序转移到了内核中。

## 关于 io_uring 的补充

- **零拷贝。** 要实现真正的零拷贝 I/O，需要提前使用 `io_uring_register_buffers()` 注册你的缓冲区——这避免了内核在每次操作时重新映射内存。对于网络发送，特别关注 `IORING_OP_SEND_ZC`（需要内核 6.0+），它完全跳过了将缓冲区复制到内核的步骤。
- **SQPOLL 消耗 CPU。** 即使你的队列为空，`IORING_SETUP_SQPOLL` 也会保持一个内核线程不断旋转和轮询，这会消耗 CPU。有一个空闲超时（`sq_thread_idle`），超时后它会退回到休眠状态，但这并不是免费的。
- **异步错误处理。** 错误作为 cqe 的 `res` 字段的一部分异步返回（并且必须异步处理）——而不是像普通同步系统调用那样作为直接返回值。

## 总结

io_uring 是现代 Linux 世界中异步 I/O 的新标准，老实说，在支持它的系统上，我看不到太多还需要使用 epoll 的理由。对于在现代 Linux 服务器上从头开始的项目，比如我们的 TinyGate 重写，io_uring 绝对是正确的选择。我是一个坚定的"在合理的时候尽快放弃对旧系统支持"的拥护者——如果你还在运行一个 7 年前发布的内核，在我看来，这不是一个好主意……
